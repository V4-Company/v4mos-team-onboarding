O que e o V4MOS?

V4 Marketing Operating System (V4MOS) — SaaS para diagnostico de marketing digital, gestao de workspaces, integracoes com plataformas de anuncios e assistente de IA. Feito para agencias e times de marketing digital.

Target

Agencias e times de marketing digital que precisam de diagnosticos, gestao centralizada e inteligencia de dados para suas campanhas.

Problema

Diagnosticos manuais, dados fragmentados em dezenas de plataformas, sem IA para escalar analises. Decisoes baseadas em instinto, nao em dados.

Estrutura

Arquitetura multi-tenant em tres niveis: Organization (empresa) → Workspace (cliente/projeto) → Users (membros com roles).

Features

12 modulos principais que compõem a plataforma V4MOS.

backend

Diagnostico Digital

Diagnostico de marketing com IA: steps guiados, perfil de marketing, questionarios adaptativos e geracao de artefatos.

ai

Chat IA

10 agentes especializados, 40+ ferramentas, suporte a GPT/Claude/Gemini, streaming via SSE, historico e contexto de workspace.

backend

Integracoes

130+ tipos: Facebook Ads, Google Ads, TikTok, LinkedIn, Instagram, Shopify, VTEX, Kommo, Bitrix, Pipedrive, RD Station e mais.

frontend

Painel Analitico

Dashboard com metricas de performance, graficos, funil de marketing, commitment tracking e visualizacoes por plataforma.

backend

Gestao de Membros

Convites por email e link, roles admin/editor/viewer, service accounts para integracao M2M, gestao de organizacoes.

backend

Planos e Assinaturas

Gestao de planos, assinaturas, pagamentos via Eduzz. Webhooks para eventos de cobranca, trial, upgrade e cancelamento.

ai

Deep Research

Pesquisa profunda por CNPJ com IA: analisa dados publicos, competidores, presenca digital e gera relatorio completo da empresa.

ai

Artefatos

Artefatos de marketing gerados por IA: planos de acao, copys, estrategias, briefings e documentos prontos para uso.

ai

Skills

Templates de prompts reutilizaveis para o chat. Crie, gerencie e use skills via slash commands (/skill-name) no chat IA.

frontend

Onboarding Flow

Fluxo guiado de configuracao do workspace: web scraping do site do cliente, preenchimento automatico de perfil de marketing.

frontend

Admin Panel

Painel administrativo com feature flags, gestao de usuarios, monitoramento de plataforma e configuracoes avancadas de sistema.

backend

Anomaly Detection

Deteccao de anomalias em metricas de campanhas: eventos com severidade, limiares configurados e alertas automaticos.

authorizer

AWS Lambda authorizer — valida JWT via Identity Service e verifica membership no workspace.

Lambda
1 handler1 use caseJWT + service account validation

Valida Authorization: Bearer <token> via Identity Service para usuarios regulares. Service accounts usam x-client-id + x-client-secret. Extrai workspace_id de 3 lugares: header x-workspace-id, path param /:workspaceId e query string ?workspaceId=. Retorna IAM Policy Allow/Deny para o API Gateway.

Convencoes

Padroes e convencoes adotados no codebase. Seguir sempre.

1

Clean Architecture + DDD

Fluxo de dependencia: domain → app → infra. Domain nao importa app nem infra. App nao importa infra. Plugin ESLint (eslint-ddd-plugin.mjs) enforces em lint time.

2

Inversify DI

Todas as dependencias injetadas via container Inversify. Wiring em infra/di/containers.ts. Tokens em infra/di/types.ts. Usar constructor injection com @inject(TYPES.Foo).

3

Zod Validation

Input DTOs validam com Zod schemas. Usar Zod v4 via import { z } from 'zod/v4'. Nao instalar zod@4 separadamente — usar o override zod@3.25.67 com zod/v4 subpath.

4

Decorators

Controllers usam: @Route('method', '/path'), @Auth(), @RequireFeature('name'), @ApiTag('Tag'). OpenAPI via @ApiBodySchema, @ApiResponseSchema, @ApiQueryParamsSchema.

5

Co-located Tests

Testes ao lado do codigo: *.spec.ts para backend (Vitest), *.test.ts para frontend (Jest). Mock factories em domain/test/mocks/.

6

Prisma + Kysely

Prisma somente para migrations (pnpm migrate). Kysely para queries em runtime. prisma-kysely gera types. Nunca usar Prisma Client para queries de producao.

7

Events

Domain events raised em aggregates via this.raiseEvent(). Publicados via SQS (batch, alta volume) ou RabbitMQ (single, tempo real). Handlers processam efeitos colaterais.

8

Commit Messages

Prefixo: feat: / fix: / chore: / refactor: / docs:. Em ingles. Descritivo. Exemplo: feat: add workspace invite flow with role assignment.

9

Build lib first

Sempre rodar pnpm build:lib antes de qualquer servico. O @v4-company/lib precisa estar compilado para o TypeScript resolver os imports.

10

Repository Pattern

Interfaces de repository em domain/repositories/. Implementacoes Kysely em infra/database/kysely/. Nunca acessar DB direto de controllers ou use cases.

11

Mappers

PersistenceMapper para entity ↔ DB row. ResponseMapper para entity ↔ DTO de resposta. Toda transformacao passa por mapper — nunca inline.

12

Feature Flags

Feature flags via @RequireFeature('flag-name') no controller. Flags gerenciadas no Admin Panel. Middleware bloqueia acesso se feature desabilitada para o workspace.

13

UnitOfWork Pattern

Para operacoes transacionais, usar KyselyUnitOfWork do lib. Garante atomicidade: todos os repos dentro do UoW compartilham a mesma transacao Postgres.

14

chat sem domain layer

O servico chat usa arquitetura simplificada de 2 camadas: app e infra. Nao criar domain layer no chat — nao ha entidades de dominio complexas nesse servico.

13

Gotchas

Armadilhas comuns. Leia antes de comecar.

!

pnpm build:lib ANTES de tudo

Se voce ver Module not found: @v4-company/lib — e porque nao rodou pnpm build:lib. Rode sempre apos dar pull ou instalar deps.

pnpm build:lib # SEMPRE primeiro
!

chat NAO tem domain layer

O servico packages/chat usa arquitetura simplificada: apenas app/ e infra/. Nao crie pasta domain/ no chat. Nao ha aggregates, value objects nem domain events nesse servico.

!

Zod v4 via zod@3.25 com subpath import

Em novo codigo, importe Zod assim:

import { z } from 'zod/v4' // correto — Zod v4 import { z } from 'zod' // antigo — e v3

Nao instale zod@4 separadamente — causara conflito. O monorepo usa override zod@3.25.67 com o subpath zod/v4.

!

Docker obrigatorio para dev:prepare

O comando pnpm dev:prepare precisa do Docker rodando para subir PostgreSQL, LocalStack (S3/SQS) e RabbitMQ. Sem Docker, as migrations vao falhar e os servicos nao sobem.

!

ESLint DDD enforces layer imports

Nunca importe infra a partir de domain ou app — o ESLint vai bloquear o build. Regras em eslint-ddd-plugin.mjs. Se o lint falhar por import incorreto, mova o codigo para a camada correta.

!

Prisma e APENAS para migrations

Nunca use Prisma Client para queries em runtime. Use Kysely. O Prisma Client nem e instalado como dep de runtime — serve apenas para prisma generate e prisma migrate.

!

Repos separados usam package managers diferentes

v4-marketing-api usa npm (nao pnpm). v4-marketing-data usa yarn + uv (Python). Nao misture — o pnpm nao gerencia esses repos.

!

LangSmith Hub: prompts carregados em runtime

O chat carrega prompts do LangSmith Hub em runtime. Se falhar ao iniciar, verifique LANGSMITH_API_KEY no .env. Sem a key, o agent nao consegue carregar o system prompt.

!

workspace_id em 3 lugares diferentes

O Authorizer Lambda extrai o workspace_id de 3 fontes: header x-workspace-id, path param /workspaces/:workspaceId e query string ?workspaceId=. Seja consistente ao construir requests.

!

Service accounts: headers especiais

Service accounts nao usam Authorization: Bearer. Usam x-client-id e x-client-secret como headers. Se integrar M2M, use esses headers em vez de token JWT.

!

AWS credentials para v4-marketing-data

O v4-marketing-data acessa AWS diretamente (S3, Lambda, ECS, Aurora). Precisa de credenciais AWS configuradas no ambiente. Peca ao tech lead o perfil AWS correto.

Visao do Produto

O que estamos construindo e por que

Antes de mergulhar no codigo, e fundamental entender a visao do produto e do squad. O V4MOS.AI diagnostica qual das 7 etapas do ciclo de receita esta travando o negocio do cliente. Nosso squad e responsavel pelas tres primeiras travas: Exposicao, Atencao e Interesse.

Ciclo de Receita — 7 Etapas

Exposicao

"Ninguem ve."

O cliente investe em midia mas nao sabe se esta investindo certo. Sem benchmark, sem diagnostico de canal.

Atencao

"Veem mas nao param."

O trafego chega mas nao engaja. Criativos cansados, copy generica, sem processo de teste.

Interesse

"Param mas nao agem."

O clique acontece mas a conversao nao vem. LP ruim, formulario longo, CTA fraco.

O objetivo nos proximos 90 dias e fechar o ciclo: o cliente entra, descobre a trava, recebe um plano de acao concreto, e acompanha se esta destravando. O documento completo detalha contexto, horizontes, riscos, decisoes abertas e estado atual de cada projeto.

Roadmap do Produto

O que e um Roadmap?

Um roadmap de produto e o mapa estrategico que conecta a visao do produto com a execucao do time. Ele responde tres perguntas fundamentais:

01

Para onde vamos?

Define a direcao estrategica e os objetivos que o produto precisa atingir.

02

O que vamos construir?

Prioriza as entregas em horizontes de tempo — o que vem primeiro e por que.

03

Por que isso importa?

Conecta cada entrega ao impacto no negocio, no cliente e na receita.

No V4MOS, o roadmap e dividido em horizontes de 90 dias. Cada horizonte tem um objetivo claro, entregas definidas, riscos mapeados e criterios de sucesso. Isso permite que o time saiba exatamente o que esta construindo, por que, e como medir se deu certo.

Como dev, entender o roadmap te ajuda a tomar decisoes melhores no dia a dia: saber por que uma feature existe, qual a prioridade real, e como seu trabalho se conecta ao objetivo maior do produto.
